# `3.35.1` ‚Üí `3.35.2`

## Bug 0010 (P1) ‚Äî `phase_bootstrap` recorded `${PPID}` instead of the claude-session PID

The Story-152 mechanical `startup.sh` boot path wrote each agent's tracker
(`implementations/.agents/<id>.json`) with `claude_pid="${PPID:-0}"`. `$PPID`
is the immediate parent of the transient bash that ran `phase_bootstrap` inside
`startup.sh`'s own pipeline ‚Äî a process that has already exited by the next boot.
The READ side (`wow-existing-agent-id.sh` ‚Üí `wow_find_claude_pid`) walks the full
process ancestry to the `claude` session process, so the writer and the reader
computed different PIDs and the tracker **never resolved** on a subsequent boot.

Impact (every role, every `startup.sh` boot):

- Story 121's idempotent agent-id resolution was re-opened ‚Äî each reset/re-boot
  minted a fresh agent ID and orphaned the prior tracker (ID drift).
- The liveness / idle-monitor `role ‚Üí claude_pid ‚Üí activity-row` mapping
  (stories 108/110/111) keyed on a dead PID, so a freshly-booted peer could be
  mis-classified by `idle-monitor.py`.

### Fix

`phase_bootstrap.sh` step 3 now captures the canonical session PID:

```sh
claude_pid="$(wow_find_claude_pid 2>/dev/null || echo 0)"
```

`whats-my-role.sh` is already sourced at step 2, so `wow_find_claude_pid` is in
scope; its `echo`-PID / `return 1` contract makes the `|| echo 0` fallback fire
when no `claude` ancestor exists (e.g. a detached launch). Writer and reader now
walk to the SAME `claude` ancestor ‚Äî symmetric ‚Üí the tracker resolves on the next
boot.

### Added

- `plugin/tests/bootstrap-tracker-claude-pid-roundtrip.sh` ‚Äî BEHAVIORAL
  producer‚Üíconsumer round-trip (not a source grep): drives the real
  `phase_bootstrap` under a graft `ps`-shim (only `ps` is stubbed; the synthetic
  `node‚Üíclaude` ancestry is grafted onto the test-body PID and the real `ps` is
  delegated for every other PID, so the test is deterministic even when the suite
  itself runs inside a real `claude` session). Asserts (a) the recorded
  `claude_pid` equals the stubbed claude PID, (b) `wow-existing-agent-id.sh`
  then resolves the just-written tracker, and (c) the no-claude-ancestor path
  records `0`.

### Modified

- `plugin/scripts/startup/phase_bootstrap.sh` ‚Äî `${PPID:-0}` ‚Üí `wow_find_claude_pid`.

Suite count: 199 ‚Üí 200 plugin suites.

### Consumer action

`/reload-plugins` + restart peers. The fix applies on next `startup.sh` boot;
existing sessions keep their current (possibly dead-PID) tracker until they
re-boot. After upgrading, a fresh boot's tracker will carry the live session PID
and `wow-existing-agent-id.sh` will resolve it idempotently across resets.
