## 148

## Modified

- **`plugin/scripts/hooks/wow-forbid-direct-bus-write.sh`** ‚Äî split the single over-greedy regex into a precise redirect pattern (`(>>?)[[:space:]]*[^[:space:]]*<bus>` ‚Äî bus must immediately follow `>` or `>>`) and a separate pattern for the write-only tools `tee` / `sed -i`. Reads in any position ‚Äî `cat <bus>`, `jq ‚Ä¶ <bus> | tail`, multi-statement commands with an unrelated redirect ‚Äî are no longer blocked.
- **`plugin/tests/forbid-direct-bus-write.sh`** ‚Äî adds 4 allow-cases including the red-green demonstrator (a read after an unrelated `>` redirect, which the old greedy regex blocked).

## Why

The previous `(>>|>|tee|sed -i)[[:space:]].*<bus>` matched any unrelated redirect that merely preceded a bus path in the same command, turning legitimate reads into denied tool calls. Reads are now reliably allowed; writes are still blocked.

`/reload-plugins` + restart peers per standard upgrade hygiene.

## Modified

- **`plugin/mcp/claude-wow-server/server.py`** ‚Äî `handle_bus_emit` adds a liveness check: when `to` is an exact agent ID (not `*` / not `<role>-*`), the server requires `implementations/.agents/<to>.json` to exist. A fabricated or dead ID returns a JSON-RPC error naming the ID and suggesting a role-glob, instead of silently appending a message no agent receives.
- **`plugin/tests/bus-emit-rejects-dead-agent-id.sh`** ‚Äî new regression: dead ID ‚Üí error + no append; broadcast/glob/live-exact ID ‚Üí ok.
- **`plugin/tests/mcp-server-learnings-auto-inject.sh`** ‚Äî collateral fix: the compaction-occurred case now creates a tracker file for its exact-ID receiver, mirroring real usage (the PostCompact hook fires while the agent is alive).

## Why

`validate_to()` previously checked only the shape of an exact ID; a fabricated or dead ID passed validation, got written, and was silently dropped by every bus-tail filter ‚Äî the message reached no one. CC now sees the JSON-RPC error and can re-emit to the role-glob.

- **`plugin/commands/_manager-startup.md`** ‚Äî Phase 1 step 2b is a new team-identity claim flow (read `.my-team` or pick an unclaimed name, append to `team_names_repo.jsonl`, push). Step 7 feat-branch cleanup is scoped to `feat/$TEAM/`. Phase 3 step 7 surveys other active teams and reports them.
- **`plugin/commands/manager.md`** ‚Äî story-creation flow uses `feat/$TEAM/<NNN-slug>`; worktree path drops the team segment. Story template adds `<!-- team: $TEAM -->` on line 2. Backlog template carries the same marker. Standing-authority commits add a `WOW-Team: $TEAM` trailer. Sprint dispatch + stacked-child branch creation use `feat/$TEAM/`. PR-nudge carries the title prefix `[$TEAM]` and the trailer convention.
- **`plugin/commands/_agent-protocol.md`** ‚Äî file-layout shows `team_names_repo.jsonl` (tracked) + `.my-team` (gitignored). New "Team scoping" subsection documents branches, PRs, commits, story/backlog marker, worktree-derive, and team identity end-to-end.
- **`plugin/commands/senior-developer.md`** ‚Äî reads `TEAM` from `.my-team`; pushes `feat/$TEAM/<NNN-slug>`; PR title prefixed `[$TEAM] feat: <title>`; commit trailer `WOW-Team: $TEAM` alongside `Co-Authored-By`.
- **Consumer project (this repo, dogfood)** ‚Äî `.gitignore` ignores `/implementations/.my-team`; this clone joins as team `falcon`.

## Why

Multiple WOW teams run against one project as separate clones sharing only the GitHub remote. They previously clashed and got confused by each other's branches, PRs, commits, and pushed stories. Teams now coexist on the same repo like multiple humans would: each clone knows its team (`.my-team`), claims a name in the shared registry (collision-detected via git push), and labels all shared-remote artifacts. The bus, `.agents/`, worktrees, and agent IDs remain per-clone ‚Äî they never share runtime state.

## Consumer action
