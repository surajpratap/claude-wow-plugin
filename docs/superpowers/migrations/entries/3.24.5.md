## Modified

- **`plugin/scripts/wow-process/idle-monitor.sh`** — at spawn time, a new `sweep_project_orphans` function runs **before** the existing single-PID lock check. It kills any prior `idle-monitor.py` process whose argv carries this project's `--project <PROJECT_DIR>` CLI tag. The python invocation in the while loop now passes `--project "$PROJECT_DIR"` as a process-table marker that the sweep matches via `pgrep -f`. Cross-project scoping is enforced by the literal-path match — other projects' monitors are never touched. Idempotent zero-match no-op when no orphans exist.

## Why

M observed **93 live `idle-monitor.py` orphans** on the dev machine on 2026-05-19, accumulated across sessions / plugin versions in one project. The single-PID lock only catches a *second concurrent spawn within one session*; it can't sweep prior-session orphans whose wrapper was SIGKILL'd without its EXIT trap firing. Every orphan kept polling `.activity.jsonl` every 60s and emitting nudges to M's bus-tail — 93× the intended rate, plus 93 idle Python processes' RAM. Worse, fixes shipped in stories 098/099/110/111/129 never reached the *running* monitor — M was being nudged by pre-098 logic the whole week.

The backlog's "exec drops trap" hypothesis was disconfirmed at plan-review: the current wrapper uses a `while; python3 …; done` foreground loop (no `exec`). The actual leak path is `kill -KILL wrapper → python reparented to PPID=1 → still alive`. The sweep handles this mechanically: each new wrapper sweeps before spawning.

## Added

- **`plugin/tests/idle-monitor-cross-session-sweep.sh`** — pins the new behaviour mechanically: (a) happy path — orphan a python by SIGKILLing the wrapper, respawn, assert exactly one python alive; (b) cross-project scoping — two projects' orphans, respawn A's wrapper, assert A swept but B preserved; (c) anti-revert — `idle-monitor.sh` body still contains the sweep function definition + invocation + the `--project` CLI tag on the python invocation. Any future refactor that silently drops the sweep fails the test.

## Consumer action

`/reload-plugins` + restart peers per standard upgrade hygiene. On next M startup, the idle-monitor wrapper sweeps stray orphans automatically — no manual `pkill` cleanup is needed even on machines with accumulated leaks from older versions.

## Why not a full root-cause fix (PR_SET_PDEATHSIG / parent-PID poll)

Out of scope for this story. The sweep is sufficient for the user-visible symptom (orphans get killed on the next monitor spawn, which is at most one session-start later). A parent-death-signal handler in `idle-monitor.py` is a follow-on backlog item if M wants belt-and-suspenders — it would prevent the orphan from existing at all rather than killing it post-hoc.

## Closes

Backlog item 165. Predecessors: 105 (PID-file convention), 109 (python-child respawn loop), 110 (foreign-PID skip), 111 (idle-monitor wake event).

## Modified

- **`plugin/mcp/claude-wow-server/server.py`** — captures its own source mtime at startup (`_SERVER_STARTUP_MTIME`); every `tools/call` first runs `_check_freshness()`, which returns a JSON-RPC `-32603` (Internal error) when the source has been modified on disk since startup. The error message names `/reload-plugins` explicitly. Single check point covers all current tools (`bus_emit`, `declare_idle`, `resume_work`) and any future ones automatically. Per-call overhead: one `stat()` syscall (~µs).
- **`plugin/mcp/claude-wow-server/server.py` (FINDING-36 key alignment)** — the Story-103 per-item code-review suppression read the `pr-created` payload key `pr_base`, but no producer emits that key (the canonical key is `base`), so the suppression was permanently **inert**: every per-item integration-branch PR still triggered a redundant `code-review-request`. Fixed the consumer to read `base` (aligned to the established producer key, not the other way around).
- **`plugin/commands/_agent-protocol.md` + `plugin/commands/senior-developer.md`** — document the canonical `pr-created` suppression key (`base`) so producer and consumer cannot silently diverge again. The marker phrase `suppression keys off `base`` appears in both, pinned by the regression test.

## Why

The per-item code-review skill ran ~14 times across the 2026-05-18 sprint, all returning NOT-ELIGIBLE — because every per-item PR auto-merged before the skill's eligibility gate cleared. Story 103's sprint-mode `code-review-request` auto-inject suppression was correct, but **inert**: the running MCP server process predated 103's merge, so the suppression code wasn't loaded. CC doesn't hot-reload MCP server source.

Picked brainstorm option 1 ("MCP-server reload discipline") over option 2 ("gate per-item auto-merge on code-review skill completing") because the staleness class affects every server.py change mid-sprint, not just `code-review-request`. One mtime check covers them all. Implementation form: server-side self-detection (rather than a sprint-merge-bump.sh signal) — CC has no API for peers to invoke `/reload-plugins`, so a downstream signal still requires a human-loop. The server-side check IS the loop: whoever invokes a tool next sees the staleness immediately.

## Added
