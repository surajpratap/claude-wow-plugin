## 154 ‚Äî Monitor truncation wrapper

### Added

- **`plugin/scripts/wow-process/monitor-pipe.sh`** + **`monitor-pipe.py`** ‚Äî every Monitor source (bus-tail, github-bridge, idle-monitor, slack-bridge spawn, slack-events-feed) pipes its stdout through this wrapper. It appends the full untruncated line to `${ROOT}/implementations/.monitor-events/<purpose>/<task-id>.jsonl` (fcntl.flock-serialized + fsync) and emits a short pointer (~150-200 chars) on its own stdout naming the file, the 1-indexed line number, and the MCP tool to call. The bash launcher mirrors `idle-monitor.sh`/`idle-monitor.py` precedent; Python is required because macOS lacks `flock(1)`.
- **`mcp__claude-wow__monitor_event_read`** MCP tool ‚Äî `monitor_event_read({event_file, line})` returns `{event: "<raw line>"}` on success or `{error, event_file, line}` on missing file / out-of-range line / path-safety violation. Pure read; no side effects. Path-safe via realpath comparison against `${ROOT}/implementations/.monitor-events/`.
- **`plugin/scripts/wow-process/monitor-events-trim.sh`** ‚Äî interim standalone trimmer (until story 152 ships `phase_sweep_monitor_events`). Three drop conditions: (a) file mtime >24h, (b) tracker-orphan + mtime >1h grace, (c) rmdir empty per-purpose dirs.
- **`?`-suffix conditional purpose contract** in `role-process-map.json` ‚Äî entries ending in `?` (e.g. `slack-bridge-spawn?`, `slack-events-feed?`) flag purposes whose presence is conditional on creds. Downstream consumers (`monitor-spec.sh`, `post-compact-restore.sh`) strip the `?` before invocation; the `?` is map-level only.
- **`## Reading Monitor events` doctrine preamble** in all 5 role files ‚Äî "call `monitor_event_read({event_file, line})` to load the full event; never act on the truncated pointer text alone."
- **10 new test suites** in `plugin/tests/`: `monitor-pipe-pointer-shape.sh`, `monitor-pipe-untruncated-roundtrip.sh`, `monitor-pipe-concurrent-writers.sh`, `mcp-server-monitor-event-read.sh`, `monitor-events-trim.sh`, `role-process-map-coverage.sh`, `bus-tail-with-pointer.sh`, `github-bridge-with-pointer.sh`, `idle-monitor-with-pointer.sh`, `tester-md-cleanup.sh`. **Suite count: was N (pre-154), now N+10.**

### Why

CC's Monitor tool truncates each event notification at ~500 chars. Long bus messages (sprint manifests, retro stats, plan payloads), PR review bodies, and Slack message threads were silently chopped ‚Äî the consuming role saw a fragment with no path back to the original (bus inode swaps invalidate offsets; bridge stdout is gone). The wrapper persists full events to disk; the MCP tool fetches them on demand. Mechanical, deterministic, no doctrine to memorize ‚Äî the pointer line literally names the tool to call.

### Consumer action

`/reload-plugins` + restart all peers per standard upgrade hygiene. Each newly-armed Monitor in the next session goes through the new wrapper. Old monitor events from the prior session are lost (they were already truncated; no recovery possible).

For projects with the Slack bridge enabled, M's startup sweep + the new `monitor-events-trim.sh` (cron-or-startup) drop stale event files automatically.
