## 168 — GitHub bridge: surface per-PR sub-call failures into the degraded signal

Per-PR sub-call failures (reviews / issue-comments / review-comments / check-suites)
were swallowed by `except Exception: pass`, so a broken/auth-failing bridge silently
stopped delivering review/comment/CI events with no signal. They now feed the same
`failure_counts` / `DEGRADATION_THRESHOLD` machinery the list call uses → a
`bridge-status: degraded` (naming the failing endpoint) after the threshold. A
legitimately-empty result is NOT a failure. Rate-limit sub-call failures continue to
THROTTLE (story 167), not degrade. Final item of the rate-limit sprint stack.

### Added
- `plugin/tests/github-bridge-subcall-degraded.sh` — behavioral, threshold-respecting test.

### Modified
- `plugin/bridge/github/run.py` — `_emit_review_events` / `_emit_comment_events` /
  `_emit_ci_events` return per-endpoint non-rate-limit failures; the poll loop feeds
  them into the degradation accounting (reset/recovered moved to a gated end-of-cycle).

### Note
- A `bridge-status: armed` can now carry one of two reasons: `recovered: <repo>` (168 —
  a degraded repo's cycle went clean) or `recovered: cadence shrinking toward floor`
  (167 — adaptive poll interval shrinking). Consumers keying on recovery should match
  the `reason`, not `state == "armed"` alone.

Suite count: +1 plugin suite.

### Consumer action
`/reload-plugins` + restart peers. Effective on the bridge's next poll cycle.
