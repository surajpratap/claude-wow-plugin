## 156 тАФ Slacker tracks human interactors

The bridge now keeps a per-project registry of every Slack user who has interacted with the bot. S uses the resulting `interactor` field on each forwarded event to adapt reply vocabulary (plain-English for non-technical interactors, technical jargon OK for engineers) and to enrich `question` payloads routed to M with a `from_interactor` object.

### Added

- `plugin/bridge/slack/src/bridge/interactors.ts` тАФ `Interactors` class plus pure helpers `classifyTechnicality(title)` and `parseOverrides(path)`. Class methods: `ensureInteractor(client, userId)`, `getRaw(userId)`, plus a `_seedForTest` test seam.
- `plugin/bridge/slack/tests/interactors.test.ts` тАФ node:test unit suite (12 cases) covering the classification matrix, override-block parsing, first-contact/repeat/TTL behavior, persistence + override-merge.
- `plugin/tests/slacker-interactors-{schema,first-contact,repeat-visit,ttl-refresh}.sh`, `plugin/tests/slacker-{classification,mention-form,interactor-overrides,question-to-m-includes-interactor}.sh` тАФ 8 new bash tests. Suite count: 145 тЖТ 153.
- `plugin/commands/slacker.md` `## Human-interactor registry` section тАФ storage layout, override format, vocabulary-mapping cheat-sheet, mention-form rule, lookup-on-reply pattern.

### Modified

- `plugin/bridge/slack/src/bridge/required-scopes.ts` тАФ adds `users:read.email` to `REQUIRED_SCOPES`. **Consumers MUST reinstall the Slack app** for the new scope to take effect on the bot token; otherwise the bridge's startup preflight rejects the token as missing the scope. See `plugin/bridge/slack/README.md` scope table.
- `plugin/bridge/slack/src/bridge/handlers.ts` тАФ `registerHandlers({ ..., interactors })` arg added. Each inbound feed-write path enriches the event with an `interactor: InteractorRecord | null` field.
- `plugin/bridge/slack/src/bridge/http-server.ts` тАФ `HttpContext` gains `interactors` field (HTTP routes don't use it yet; passed for symmetry).
- `plugin/bridge/slack/src/index.ts` тАФ instantiates `Interactors` from `WOW_INTERACTORS_PATH`, `WOW_INTERACTOR_OVERRIDES_PATH`, `WOW_INTERACTOR_PROFILE_TTL_DAYS` env vars (registry no-ops when path unset). Passes the instance into both `registerHandlers` and the `ServerContext`.
- `plugin/bridge/slack/package.json` тАФ `test` script globs `tests/*.test.ts` (was `tests/smoke.test.ts` only) so the new unit suite runs.
- `plugin/commands/slacker.md` step 5 spawn command тАФ derives `WOW_INTERACTORS_PATH` from `git rev-parse --show-toplevel`'s slash-to-underscore project key, sets it + `WOW_INTERACTOR_OVERRIDES_PATH=$ROOT/implementations/learnings/slacker.md` on every spawn variant. The `## Escalation flow` payload shape adds `from_interactor`.

### Consumer action

1. `/reload-plugins`.
2. **Reinstall the Slack app to your workspace** (Slack OAuth dashboard тЖТ Reinstall to Workspace) so the bot token picks up the new `users:read.email` scope. Without this, the bridge's preflight rejects the token at startup.
3. Restart `S`.

The new registry is opt-in: if you don't want it on this project, set `WOW_INTERACTORS_PATH=""` in `slacker.md` step 5 (or skip the reinstall) and the bridge runs in legacy mode тАФ `interactor: null` on every feed event, no per-user adaptation, but everything else continues to work.
